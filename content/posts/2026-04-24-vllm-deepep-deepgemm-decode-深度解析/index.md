  <h3 class="prologue-h3" id="pr-s1">① Decode 阶段的核心特点</h3>
  <p>Decode 阶段每步只新增 1 个 token，整个 batch 也就几十到两百。三条硬约束决定了后端选型：</p>
  <ul>
    <li><b>延迟敏感</b>：ITL 目标 &lt; 30-50 ms / token，不能像 Prefill 那样忍受 20 ms/层的 MoE。</li>
    <li><b>形状固定</b>：必须能进 <b>CUDA Graph</b>，否则 launch 开销就会吃掉毫秒。</li>
    <li><b>RDMA latency 主导</b>：带宽不是瓶颈（batch 小），发起 RDMA 请求的固定开销才是。</li>
  </ul>

<div class="layer-banner" id="layer1">
  <div class="tag">Layer 1</div>
  <h2 class="t">问题：为什么 Prefill 的 HT 路径不能直接用于 Decode</h2>
  <div class="s">SM 竞争、动态 shape、RDMA latency 都在砸 Decode 的延迟目标</div>
</div>

<div class="formula-box std-box">
<div class="formula-label">❌ HT 路径拿到 Decode 会发生什么</div>
<ol>
  <li><b>动态 shape</b>：<code>num_tokens_per_rank</code> 每步都不一样 → 无法 CUDA Graph。</li>
  <li><b>SM 抢占</b>：HT 通信用 20 SM，Decode 本来 SM 就空不下来。</li>
  <li><b>Contiguous 排列</b>：每步都 scatter 一遍，把 Decode 的微小开销再放大。</li>
  <li><b>BF16 combine 太重</b>：batch 小时带宽并非瓶颈，但 2× 数据量意味着 2× 发起开销。</li>
</ol>
</div>

<div class="formula-box sm-box">
<div class="formula-label">✅ LL 路径的对应解法</div>
<ol>
  <li><code>low_latency_mode=True</code> + cooperative launch kernel，形状固定。</li>
  <li><code>num_sms=0</code> — 完全靠 RDMA 硬件自走。</li>
  <li>Masked layout — 每 expert 都按 max_m 占位，kernel 比较向量不读每行。</li>
  <li>Combine 内置 weighted-sum，每 token 一次聚合。</li>
</ol>
</div>

</div>

<p>LL 模式下 <code>DeepEPLLAll2AllManager</code> 的几个关键 knob：</p>
<ul>
  <li><code>num_qps_per_rank = 64</code>：RDMA 的尾延迟（tail latency）可以通过多 QP 并发分摊。</li>
  <li><code>allow_nvlink_for_ll = True</code>：同 Node 的 GPU 仍然走 NVLink 节省跳数。</li>
  <li><code>low_latency_mode = True</code>：DeepEP 使用 cooperative kernel launch（<code>cudaLaunchCooperativeKernel</code>），可以写入远端 GPU 内存的 LL staging。</li>
  <li><code>num_sms = 0</code>：通信不占算力 SM，Decode GEMM 可以用满 132 SM。</li>
</ul>

<div class="deep-dive">
<span class="dd-label">DEEP DIVE</span>
<strong>为什么 num_sms=0 还能跑 RDMA？</strong>
<p>RDMA 的数据路径由 IB HCA 和 InfiniBand 交换机完成，SM 只是用来<b>触发</b> work request（例如 <code>ibv_post_send</code>）和轮询 completion。LL kernel 把这些 SM 工作量压到最小：每个 rank 用固定几个 warp 发起 send，剩下的 SM 自由给 DeepGEMM 用。HT 模式留 20 SM 是为了把 token 先搬一步做 layout，LL 直接让 token 留在原位——就是不需要这些 SM。</p>
</div>

<div class="layer-banner" id="layer3">
  <div class="tag">Layer 3</div>
  <h2 class="t">DeepGEMM Masked：(E, max_m, K) 3D tensor</h2>
  <div class="s">代价是 padding，换的是 CUDA Graph 与省掉 permute</div>
</div>

<p>接口是 <code>fp8_m_grouped_gemm_nt_masked(A, B, C, expert_num_tokens)</code>：</p>
<ul>
  <li><code>A: (E, max_m, K)</code> FP8，每个 expert 的有效部分紧贴前面，后面 padding 不读。</li>
  <li><code>B: (E, N, K)</code> FP8，所有 expert 的权重连在一起（和 Contiguous 相同）。</li>
  <li><code>expert_num_tokens: (E,)</code> int32，kernel 每到新 expert 读一次判断。</li>
  <li><code>C: (E, max_m, N)</code> BF16，padding 区域不写。</li>
</ul>

<div class="deep-dive">
<span class="dd-label">DEEP DIVE</span>
<strong>Masked 与 Contiguous 的 kernel 层差别</strong>
<p>Contiguous 读 <code>m_indices[row]</code> 判 expert；Masked 读 <code>expert_num_tokens[cur_expert]</code> 判 tile 是否越界。前者省内存，后者省一次 ep_scatter。对 Decode 来说，ep_scatter 的 atomic_add 哪怕只占几十微秒也是纯开销，Masked 直接绕过去。另外 Masked 的 expert_num_tokens 是<b>指针</b>而不是值，可以在 CUDA Graph 录制后运行时改——否则 Graph 就白录了。</p>
</div>
